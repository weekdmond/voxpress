## 二、产品定位（核心，必须先看）

**SpeechFolio 是个人工具，不是 B2B SaaS。**

两条具体路径：

1. **个人学习工具**：把感兴趣的公开视频转成可读、可搜索、可标注的文章用于自学
2. **博主自归档**：创作者把自己的口播视频转成公众号 / 知乎等渠道的文字稿

**不是**：客户交付产品 / 企业内容生产中台 / MCN 内容农场工具。

> ⚠️ **重要**：这一前提推翻了本轮讨论里很多评论。如果你（或未来的我）发现自己又在用"客户演示就绪度 / 交付感 / 企业级"评价产品，请先停下来回到这一节。
>
> 这个定位也大幅降低了若干"看起来很严重"的风险（见第三节）。

## 三、产品风险评估（按定位重新排序后）

完整的 R1–R10 是按 B2B 视角列的。在"个人学习 + 博主自归档"前提下，重要性大幅变化：

### 仍然成立

- **R7 运维脆弱度**：cookie 过期、单机部署、Postgres 单点。自己用也会被打断
- **R8 数据安全**：抖音 cookie 在 DB 里建议加密；备份策略
- **R9 模型依赖**：DashScope 单点。中期加 LLM provider 抽象（OpenAI / Claude / 火山）
- **R2-子集 抖音反爬**：cookie 维护成本递增；中期必须做多平台（YouTube 官方 API 是低风险解）

### 大幅降权（B2B 视角才严重）

- ~~R1 著作权~~：个人学习落入合理使用；博主自归档是自有 IP；几乎没问题
- ~~R3 AIGC 备案~~：不对外服务就不用备案
- ~~R4 用户画像模糊~~：已澄清
- ~~R5 成本结构对定价敏感~~：不商业化就没这事
- ~~R6 同质化竞争~~：自己用的工具不需要打竞品
- ~~R10 品牌壁垒~~：同上

### 上线前还是建议修掉的工程 Blocker

来自 `REVIEW_v2.md`，与定位无关，纯属代码级风险：

- **B1 SSRF**：`voxpress/routers/media.py:46` 的 `httpx.AsyncClient(follow_redirects=True)`
- **B5 SSE 静默丢事件**：`voxpress/sse.py:54` 的 `Queue(maxsize=512)` + `put_nowait` + `except Exception: return` 会让前端进度卡住

## 六、文章质量调优方法论（暂不执行，方法论沉淀）

之前 Claude Code 给过的结论"模型限制，prompt 调优提升不大"被判定为**伪结论**——理由：当时没有 eval set，所谓"提升不大"等于"在 3 个例子上凭感觉看起来差不多"。

判断"是不是真到模型上限"的唯一方法：**同一 prompt 换更强模型（qwen-max / Claude Sonnet 4.6）跑同一 eval set**。如果更强模型明显好 → 是 prompt 设计问题；如果都差不多差 → 才是组合到顶。

### 真正的流程

**Phase 1：建 eval set（2 工日，没有它后面全是假的）**

8–10 个 case，覆盖不同长度 / 主题 / 当前质量基线。每个 case 准备：

- 输入：原始 ASR transcript
- 参考输出：你心目中"好"的版本（手改过的 / 竞品的 / qwen-max 出的高质量版）
- 3 档 label（优 / 中 / 差）+ 1–2 句 note（**为什么这样标，写具体的失败模式而不是"质量不行"**）

5 个评分维度：信息保真 / 结构 / 可读性 / 简洁度 / 风格

**Phase 2：基线诊断（1 工日）**

跑当前 prompt 拿到基线分。**找系统性失败模式**（永远丢例子 / 标题永远像通稿 / 长视频降得明显 / ...），不是为了拿数字。

**Phase 3：定向迭代（2-3 工日）**

按 ROI 从高到低试杠杆：

🔴 高 ROI：
1. **拆成 multi-pass**（outline → draft → polish），不要单轮。**这是质量天花板的最大杠杆**
2. **强制 JSON schema 输出**，让 LLM 必须分步思考
3. **加 few-shot**（1-2 篇你认为好的"输入→输出"对儿）
4. **预处理 ASR**（清口头禅、重复字），脏 transcript → 脏文章

🟡 中 ROI：
5. **Anti-pattern 显式列出**（"❌ 不要用'我们今天聊聊'开头" 比 "要简洁" 有效 5 倍）
6. **角色 + 任务分离**（system 写角色，user 写任务）
7. **Temperature 0.3-0.4**（organize 不需要创造性）

🟢 低 ROI：
8. max_tokens / 输入分段
9. self-critique pass

**Phase 4：A/B 对比**

每改一版 prompt → 在同一 eval set 上跑 → 把每个 case 的新旧输出 side-by-side 摆。**不要看平均分，看 trade-off 和增量**。

**Phase 5：扩样固化**

最佳版本在 8–10 case 稳定 → 扩到 50 个再跑一遍。前 10 个是过拟合温床。

### 分工

**只有用户能做的**：选 case、标"优/中/差"、写"为什么差"的 note、看 A/B 输出哪个更像想要的

**Claude 可以做的**：写 multi-pass 改造、prompt 从 settings 走、eval harness、ASR 预处理函数

## 七、Prompt-Lab 工具项目（已建，暂停使用）

### 目的

为 Phase 1-5 提供工具支撑：批量跑 case、A/B 对比、多轮 pipeline 实验。

### 位置

`/Users/auston/cowork/dy_docs/voxpress-prompt-lab/`（与 `voxpress-api/` 同级，独立项目）

### 状态

✅ 已搭好，未使用，未跑通。

包含：

- `pyproject.toml`（hatchling + uv）+ `.python-version`（3.12）
- `pl/config.py`：从 `.env` 加载 DashScope key 与默认 LLM 参数
- `pl/llm.py`：DashScope OpenAI 兼容客户端 + token / ¥成本统计
- `pl/preprocess.py`：ASR 文本清洗（口头禅、重复字）
- `pl/pipeline.py`：single_pass / multi_pass 执行器
- `pl/runner.py`：跑 cases × version，输出到 `runs/<timestamp>-<version>/`
- `pl/diff.py`：side-by-side HTML 对比（无第三方 markdown 渲染依赖）
- `pl/cli.py`：`pl run / pl diff / pl list-versions / pl list-runs / pl list-cases`
- `pl/prompts/v0_single/prompt.txt`：单轮 baseline 占位（**真用之前要替换成 voxpress-api 实际线上 prompt**）
- `pl/prompts/v1_multi/{outline,draft,polish}.txt`：三轮 multi-pass 模板
- `cases/_template.json` + `cases/README.md`

### 暂停原因

意识到 SpeechFolio 设置页（`/settings`）已经直接支持编辑 `correct` 和 `organize` 阶段的 prompt（"文章整理 Prompt"区块，"已同步"badge，可"恢复默认 / 保存"）。对**单轮 prompt 迭代**这一最常见场景，**设置页够用**。

prompt-lab 的不可替代价值仅限于：

| 能力 | 设置页 | prompt-lab |
|---|---|---|
| 单轮 prompt 编辑 + 保存生效 | ✅ | ✅（重复了）|
| 多轮 pipeline（outline → draft → polish）实验 | ❌（后端是单轮硬编码）| ✅ |
| 一次跑 8–10 个 case 不影响产线 | ❌ | ✅ |
| 同 case 上 v1.0 vs v1.1 **并排** 对比 | ❌（保存即生效，破坏现场）| ✅ |
| 每版 prompt 的 token / ¥ 统计 | ❌ | ✅ |

### 重启时怎么办（三选一）

1. **删掉 lab**，单轮 prompt 在设置页里改就行
2. **裁掉单轮，只留多轮 pipeline 实验场**，用作 voxpress-api 后端不支持的能力的实验场
3. **保留全功能**，作为长期质量基础设施

我倾向 **2**：voxpress-api 后端目前是单轮 organize，只有 lab 能验证多轮是否带来实质提升；等多轮证明有效，再回 voxpress-api 改后端 + 设置页加多轮配置。

> 决策延后，等真正回来调质量时再选。
